Authoritative description of a real Project Sanctum pipeline output, derived from run

## Top-level scopes (confirmed)

```
filesystem/      → 21 world-facing clinical .docx files (the agent-accessible world)
tasks/           → task config + grader assets (NOT world-facing)
  tasks/T1/      → task.json, prompt.md, golden_answer.docx, grader_guidance.md,

                   instructions_{prompt,golden,grader}.pdf, references/
```

A pipeline run produces both an **input-filesystem** (writer's world) and an **output-filesystem**
(post cert+repair). Filenames match across both; some files are rewritten during repair (e.g. the
two Task 7 packet files shrink after repair).

## CRITICAL RULE: "Filesystem governs" (filesystem wins)

The spec and the built world can disagree on **filenames**, by design (a "cert+repair rename").
Real example from this run:

- Spec (`structured_spec.json` → `primary_files`) lists:
  `PriorAuthClinicalPacket_10022024.docx`, `PayerMedicalNecessityCriteria_10022024.docx`
- The actual `filesystem/` files are:
  `Task 7_File A_Prior Auth.docx`, `Task 7_File B_Payer Med Necessity.docx`

**The filesystem name governs.** A spec-vs-filesystem filename mismatch is **expected drift, not a
defect.** Do **not** flag it as a missing file (Completeness) or a cross-document inconsistency.
Verify against the real `filesystem/` names.

## Trap documentation — real shape

Traps are documented out-of-world in **`meta/structured_spec.json`** under a `traps[]` array. Each
trap entry here carries:

- `affected_tasks` (e.g. `["T1"]`)
- `what_misleads`  → the **conceal surface**
- `how_agent_fails` → the failure mode
- `remediation_path` → the **reveal surface** / in-world resolution

This run documents **5 traps** for T1. Note: the field names are `what_misleads` /
`how_agent_fails` / `remediation_path` — **not** literal "trap_id" / "severity" fields, and the
exact set of spec records varies by world (here: `structured_spec.json`, `world_spec_tasks.json`,
`spec_doc.docx`; **no** `typed_spec.json` / `spec.md`). Agents must read whatever spec records are
present rather than assume fixed filenames.

## Reviewer / traceability sources (high value for verification)

Two `tasks/` files are the best ground truth for Golden Answer Traceability and Clinical Accuracy:

- **`tasks/T1/grader_guidance.md`** — Context, Golden Response, Must-haves, Common Failures,
  Acceptable Variation. Tells you exactly what a correct answer requires and the known failure
  modes (which often correspond to the traps).
- **`tasks/T1/AUDIT_README.md`** — a per-fact map of "invented vs. where to look it up," pointing
  each load-bearing fact (identifiers, doses, PD-L1, molecular status, criteria) to the specific
  `filesystem/` file and section that grounds it.

When verifying traceability or plausibility, **consult these two files** instead of guessing.
