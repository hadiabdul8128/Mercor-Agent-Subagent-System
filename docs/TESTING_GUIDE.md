# Testing, Validation & Iteration Guide

How to test the AutoQC repair pipeline, judge whether it's working, and improve it. Follow the
phases in order. Phases A–B prove the system loads and doesn't invent problems; C is the real
test (failing findings); D is how to score a run; E is how to iterate when it gets something wrong.

> Run this from inside the project folder so the agents load:
> `cd "<this repo>" && claude` then `/agents` (you should see 13).

---

## Phase A — Setup & smoke test (does it load and behave?)

1. **Open Claude Code in the project folder** (not the home dir). Run `/agents` → confirm all 13:
   intake-classifier, orchestrator, the 10 specialists, regression-agent, patch-engine.
2. **Confirm `.env` exists** (Supabase memory). If you cloned fresh from GitHub it won't be there
   (it's gitignored) — copy `.env.example` → `.env` and fill in values, or skip memory for now.
3. **Smoke test the classifier** (read-only, safe):
   ```
   /classify-autoqc
   Solution Integrity failed: tasks/T1/task.json contains a notes field reading
   "Generated by task_gen_standalone. Trap: ... correct response assigns ..."
   ```
   ✅ Expect: `folder_scope: tasks`, `writer_fixable: false`, `recommended_action: override`.
   If it says edit world files → the classifier is wrong (see Phase E).

---

## Phase B — Validate on a CLEAN world (no false positives)

Goal: prove the system does **not** invent problems in a world that already passes AutoQC.

1. Put a known-good world tree in the working dir (`filesystem/`, `tasks/`, `meta/`).
2. Run `/autoqc-repair` on a few findings you *expect to be clean / internal*.
3. ✅ Expect: clean categories report no edits; internal (`tasks/`/`meta/`) findings →
   `PASS_NO_EDIT` / override; **no world files get changed.**

This is the specificity test. A system that "fixes" a clean world is worse than useless.

---

## Phase C — Validate on REAL failing findings (the real test)

This needs a world with **actual AutoQC failures** (not the Task 7 sample, which passes).

1. Run a real world through AutoQC; collect the findings that **failed**.
2. For each failing finding:
   ```
   /autoqc-repair
   <paste the failing AutoQC finding>
   ```
3. Read the chain: classification → specialist diagnosis → **regression verdict** → patch (if PASS).
4. For a **PASS**: open `_repaired/` and read `_repaired/PATCH_LOG.md` — verify the diff is exactly
   the fix and nothing else. Confirm the original file is unchanged.
5. **Re-run AutoQC on the `_repaired/` tree.** This is the closing loop:
   - the original finding should now pass, AND
   - no *new* category should fail.

Test each verdict type at least once:
- a safe mechanical fix → **PASS** → patched in `_repaired/`
- a fix that would break a trap / another category → **FAIL** → bounced back
- a clinical-judgment change → **HUMAN_REVIEW** → listed, not applied
- internal `tasks/`/`.meta/` residue → **PASS_NO_EDIT** → no edit

---

## Phase D — Validation scorecard (judge every run)

For each finding, check:

| # | Question | Pass if… |
|---|---|---|
| 1 | Folder scope correct? | filesystem/tasks/meta labeled right |
| 2 | Routed to the right specialist? | category matches the specialist |
| 3 | Diagnosis grounded in real files? | cites actual file/section, not guesses |
| 4 | Regression gate sensible? | catches cross-category risk; doesn't over-block |
| 5 | Traps preserved? | no trap defused/labeled/weakened |
| 6 | Originals untouched? | only `_repaired/` changed (hash match) |
| 7 | Re-run AutoQC: original fixed? | the flagged finding now passes |
| 8 | Re-run AutoQC: nothing new broke? | no new category failure (the whole point) |

A run is **good** only if #8 holds — fixing the issue without creating another.

## Phase E — Iteration (when it gets something wrong)

The agents are just markdown — edit the relevant `.claude/agents/*.md` and re-test.

| Symptom | Fix in… |
|---|---|
| Wrong folder scope / mislabeled | `.claude/agents/intake-classifier.md` |
| Routed to wrong specialist | `.claude/agents/orchestrator.md` (routing table) |
| Weak/incorrect diagnosis in a category | that `.claude/agents/<category>.md` |
| Gate too strict (blocks safe fixes) or too lenient | `.claude/agents/regression-agent.md` (PASS/FAIL conditions) |
| Patch sloppy / out of scope / touched original | `.claude/agents/patch-engine.md` |
| Missed a real-world rule (e.g. filename drift) | `docs/REFERENCE_PIPELINE.md` + the relevant agent |

**Iteration loop:** edit one agent → re-run the same finding → confirm the behavior changed →
re-run Phase B (make sure you didn't break the clean-world case) → commit:
```
git add -A && git commit -m "tune <agent>: <what changed>"
git push
```

Change **one thing at a time** and re-test — same discipline the pipeline itself enforces.

---

## Honest gotchas

- **The Task 7 sample passes AutoQC** — great for Phase B (false-positive testing), useless for
  Phase C (you need real failures).
- **`.docx` auto-patching is finicky** — world files are zipped XML. Mechanical fixes to `.json`/
  `.md` are easy; complex `.docx` edits may land in HUMAN_REVIEW until the patch-engine is hardened.
- **Most clinical changes route to HUMAN_REVIEW by design** — that's correct, not a bug.
- **Memory needs `.env`** — the pipeline runs without it; only the Supabase/Obsidian recording needs it.
- **Custom agents only load in a session started inside this folder** — `/agents` confirms it.
