# Project Context — Project Sanctum

Background reading for any agent or human working in this repo.

## What Project Sanctum is

Project Sanctum produces **synthetic clinical world files**: realistic patient charts, clinical
notes, medication lists, lab results, and the task scaffolding built around them. Writers author
these world files. They are then run through **AutoQC**, an automated quality checker that flags
issues across ten categories (see `docs/AUTOQC_CRITERIA.md`).

The world files are used to build evaluation **tasks**. Each task poses a question whose correct
answer ("golden answer") is recoverable from the world files, sometimes guarded by deliberate
**traps**.

## What "world files" are

World files are the **in-world, world-facing clinical documents** — the artifacts that exist
"inside" the simulated clinical world. A clinical note, a discharge summary, a medication
administration record, a lab panel. These are what a clinician character would actually see.

Key property: world files must read as authentic clinical documents. They must not contain
out-of-world residue (generator comments, prompt text, meta-annotations), must be internally and
cross-document consistent, must be chronologically coherent, and must preserve intentional traps.

## Folder scopes: `filesystem/`, `tasks/`, `.meta/`

A flagged path's **folder scope** determines who can fix it and how.

### `filesystem/` — the world
- The world-facing clinical documents themselves.
- An AutoQC issue here is a **possible writer-fixable world-file issue**.
- This is the only scope where a writer editing a world document is normally appropriate.

### `tasks/` — task config and assets
- Task definitions, configuration, and task-specific assets (e.g. `task.json`).
- An issue here is a **task config or task asset issue**.
- Do **not** assume it requires editing world files. The problem usually lives in the task
  scaffolding, not in the clinical documents. Often handled by override or escalation.

### `.meta/` — pipeline bookkeeping
- Pipeline metadata and bookkeeping produced by the generation tooling.
- An issue here is usually **not** a writer edit. Default stance: **escalate or document**.

### `unknown` — unrecognized path
- The path does not clearly map to any scope above.
- Default stance: **ask for files or clarification** before recommending anything.

## The common AutoQC repair mistake

The recurring failure mode this whole system exists to prevent:

> A writer pastes one AutoQC issue. Claude fixes that one issue locally. AutoQC re-runs, and
> **new** issues appear in other categories — because the fix damaged cross-document consistency,
> trap architecture, chronology, or formatting.

Concretely, the mistakes are:
1. **Editing the wrong scope** — changing a `filesystem/` world file when the real problem is in
   `tasks/` config or `.meta/` bookkeeping.
2. **Normalizing intentional differences** — "standardizing" a field, date, or phrasing that was
   intentionally inconsistent for a trap or for realism.
3. **Defusing traps** — removing or simplifying something that was a deliberate trap.
4. **No regression check** — applying a fix without verifying it does not break another category.

The system counters this with: classify scope first → route to a specialist → regression-check →
copy-on-write apply.

## Worked example: the `task.json` notes residue

AutoQC reports:

> "Solution Integrity failed: `pipeline_output/tasks/T1/task.json` contains a notes field reading
> 'Generated by task_gen_standalone. Trap: ... correct response assigns ...'"

Correct classification:

| Field | Value |
|---|---|
| category | Solution Integrity |
| subcriterion | Out-of-World Residue in Document Body and Metadata |
| flagged_path | `pipeline_output/tasks/T1/task.json` |
| folder_scope | `tasks` |
| writer_fixable | `false` |
| recommended_action | `override` (override_or_escalate) |

Recommendation: **Do not edit `filesystem/` world files.** The out-of-world residue is inside
internal **task config**, not a world-facing clinical document. This is a task-scaffolding
concern; treat it as override/escalate, and preserve any trap the notes refer to rather than
acting on it.
